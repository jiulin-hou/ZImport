# Zimbra 数据导入页面 — 设计文档

| 项目 | 内容 |
|---|---|
| 文档日期 | 2026-05-20 |
| 目标系统 | Zimbra 8.8.15 FOSS @ `mail.msauto.com.cn`（`192.168.31.175`） |
| 状态 | 设计已确认，待写实现计划 |

---

## 1. 背景与目标

现有 Zimbra 自带的「导入/导出」功能在导入 tgz 时存在缺陷：当 tgz 由非标准工具打包、且内部含超长或非 ASCII 文件名时，tar 会采用 pax 扩展格式并生成 `PaxHeader` 伪条目，Zimbra 导入器无法识别，报「文件夹 /PaxHeader/ 中不能包括 document 项目类型」并导入失败。

本项目构建一个独立的 Web 导入页面，目标：

- 支持一次上传**多个 `.eml`** 文件
- 支持上传**大体积 `.tgz`**（可能超过 5GB）
- 从根本上规避 `PaxHeader` 问题
- 支持管理员批量导入与终端用户自助导入两种使用方式

### 非目标（YAGNI — 明确不做）

- 不做导出功能（只做导入）
- 不做任务优先级、资源配额、动态并发调节
- 不做用户权限管理系统（权限完全委托给 Zimbra 判定）
- 不做 `.pst` / `.mbox` 等其他格式转换（只接受 `.eml` 与 `.tgz`/`.tar.gz`）

## 2. 使用场景

| 使用者 | 场景 | 目标账户 |
|---|---|---|
| 管理员 | 员工交接、历史邮件迁移，把数据导入任意账户 | 页面上指定任意账户 |
| 终端用户 | 自助把自己的历史邮件导入自己的邮箱 | 锁定为登录账户本人 |

两种 tgz 内容都需支持：

- **纯 `.eml` 打包**：tgz 内是一批邮件文件（可能带子目录）
- **Zimbra 完整账户导出**：含文件夹结构、联系人、日历、标签，带 `meta.xml`

## 3. 总体架构

一个独立的 Python Web 服务，部署在 Zimbra 服务器本机，使用独立 venv（不污染 Zimbra 自带运行环境）。前端为单页应用，后端提供 API，后端通过 localhost 调用 Zimbra 的 SOAP / REST 接口。

```
浏览器(单页)  ──HTTPS──>  导入服务
                            ├─ web 进程   (处理登录、上传、API)
                            └─ worker 进程 (后台消费任务队列、注入邮件)
                                  │
            web    ── SOAP ──> Zimbra :7071 / :8443   (登录时验证用户身份)
            worker ── SOAP ──> Zimbra :7071           (服务账号委托认证)
            worker ── REST ──> Zimbra :8443           (注入邮件)
                                  │
                          SQLite (任务队列 + 进度状态)
```

web 进程与 worker 进程解耦，通过 SQLite 中的任务队列表通信。

## 4. 组件设计

每个组件职责单一、可独立测试。

### 4.1 前端单页

- **职责**：登录表单；文件选择（多 `.eml` 或单 `.tgz`）；管理员可填目标账户；选择目标文件夹（默认 Inbox）；文件切片；分片上传；任务进度轮询与展示；「我的任务」列表。
- **依赖**：浏览器、后端 API。
- **接口**：调用后端的登录、上传、任务查询 API。

### 4.2 认证模块

- **职责**：(a) 登录时通过 Zimbra SOAP 验证用户账号密码、区分普通用户与管理员；(b) 注入时用服务账号对目标账户做委托认证（`DelegateAuth`）取得 token。
- **依赖**：Zimbra SOAP 端点、服务账号凭据。
- **接口**：`login(username, password) -> Identity`（`Identity` 含身份类型与登录账户，不含长期 token）；`delegate_token(target_account) -> token`（worker 注入前调用）。

### 4.3 上传接收模块

- **职责**：接收分片、按分片号落盘到任务临时目录、报告已收分片（支持断点续传）、分片到齐后合并成完整文件。
- **依赖**：本地磁盘。
- **接口**：`receive_chunk(task, index, data)`；`missing_chunks(task) -> [int]`；`merge(task) -> filepath`。

### 4.4 解包归一化模块（核心）

- **职责**：解开 tgz（Python `tarfile`，原生支持 pax 格式，`PaxHeader` 对它不构成问题）；判别内容类型；产出注入模块可统一处理的输入。
- **判别规则**：包内含 `meta.xml`（或 Zimbra 导出特征结构）→ 判为「完整导出」；否则收集所有 `.eml` → 判为「纯 eml」。
- **产出**：
  - 完整导出 → 重新打包为干净的 **ustar 格式** tgz（短 ASCII 内部路径，无扩展头）
  - 纯 eml → 一个 `.eml` 文件路径列表
- **依赖**：本地磁盘。
- **接口**：`normalize(filepath) -> NormalizedInput`（`NormalizedInput` 标明类型与产出物路径）。

> 这是消灭 `PaxHeader` 问题的关键：后端始终用标准工具解包再重打包，Zimbra 永远只收到干净包。

### 4.5 注入模块

- **职责**：把归一化后的数据通过 Zimbra REST 注入目标账户。
- **行为**：
  - 纯 eml → 逐封 `POST /home/<账户>/<文件夹>?fmt=eml`
  - 完整 tgz → `POST /home/<账户>/?fmt=tgz`
- **依赖**：Zimbra REST、注入前由认证模块经服务账号委托取得的 token。
- **接口**：`inject(normalized_input, target_account, folder, token, progress_callback)`。

### 4.6 任务调度与进度模块

- **职责**：维护 SQLite 任务队列；后台 worker 按受控并发度消费；持久化每个任务的进度与成功/失败明细；提供进度查询。
- **依赖**：SQLite。
- **接口**：`enqueue(task) -> task_id`；`get_status(task_id) -> TaskStatus`；`list_tasks(account) -> [TaskStatus]`。

## 5. 端到端数据流

```
1. 登录    浏览器提交 Zimbra 账号密码 → 认证模块 SOAP 验证 → 建立 session
2. 选择    选多个 .eml 或一个 .tgz；管理员填目标账户；纯 eml 选目标文件夹（默认 Inbox）
3. 上传    前端切片（如 10MB/片）→ 逐片 POST → 上传接收模块落盘到任务临时目录
           中断可续：后端报告已收分片号，前端只补缺失片
4. 合并    分片到齐 → 合并成完整文件
5. 入队    创建任务记录写入 SQLite 队列 → 立即向前端返回 task_id，前端解绑
6. 调度    后台 worker 从队列取任务（默认并发度 1，串行）
7. 归一化  解包归一化模块解包 → 判别类型 → 产出 eml 列表或干净 ustar tgz
8. 注入    worker 经服务账号委托取得目标账户 token → 注入模块逐封 / 整包注入 REST，每步更新进度
9. 报告    前端凭 task_id 轮询任务状态 → 展示进度、成功 N / 失败 M + 失败明细
10. 清理   任务完成或失败后删除临时文件；任务记录保留 7 天后清理
```

步骤 5 是关键解耦点：上传完成即入队并返回 task_id，注入工作转入后台，用户可关闭页面、稍后凭 task_id 或「我的任务」列表回来查看进度。

目标文件夹设置仅对纯 eml 注入生效；完整导出 tgz 按包内文件夹结构原样还原，不受目标文件夹设置影响。

## 6. 认证与授权

认证分为两个互相独立的环节：**用户登录**（确认身份、决定目标账户范围）与**注入认证**（worker 实际把邮件写入 Zimbra）。两者分开是因为系统采用 web / worker 双进程架构，且注入任务可能在队列中排队较久 —— 用户登录时的 token 无法可靠地跨进程传递、也可能在排队期间过期。

### 6.1 用户登录

两种身份共用同一登录入口，后端按 Zimbra 判定结果分流：

| 身份 | 验证方式 | 可导入范围 |
|---|---|---|
| 普通用户 | SOAP `AuthRequest` → `https://localhost:8443/service/soap` | **仅登录账户本人** |
| 管理员 | admin `AuthRequest` → `https://localhost:7071/service/admin/soap` | **任意账户** |

- 登录仅用于确认身份与决定可选目标账户范围。验证完成后用户的 token **即用即弃**，不跨进程传递、不落盘。
- 管理员身份由 Zimbra 判定 —— admin SOAP 端口（7071）只有管理员账号能登录成功，后端不自行维护权限表。
- 普通用户即便在页面上手填了他人账户，后端**强制忽略并改写为登录账户**。越权在后端被拦截，不依赖前端约束。

### 6.2 注入认证（服务账号）

- 部署时配置一个专用的 Zimbra 管理员账号作为**服务账号**，其凭据存于服务器本地配置文件。
- worker 执行注入任务前，用服务账号对任务的目标账户做 `DelegateAuthRequest`，**即时取得目标账户 token、即时使用**，不落盘、不写日志。
- 这样 worker 不依赖用户登录时的 token，跨进程与 token 过期问题都被消除；SQLite 任务记录只需保存目标账户、文件路径、发起人，**不保存任何 token 或密码**。

## 7. 任务调度与并发控制

压力分析：上传阶段只写本地磁盘，不压 Zimbra；注入阶段逐封 POST 直接压 `mailboxd`、MySQL、磁盘 IO，是真正的瓶颈。服务器仅 8GB 内存，多任务并发注入会影响正常用户收发邮件。

调度方案：

- 上传与注入解耦：上传允许并发（仅写磁盘），完成后入队。
- 注入任务全部进 SQLite 队列，后台 worker **默认并发度 = 1（完全串行）**。无论多少人同时发起，Zimbra 同一时刻只承受一个导入任务的压力，压力恒定可控。
- 并发度做成**可配项**，默认 1；如需调高需显式修改配置。
- 任务状态包含「**排队中**」，前端显示「前面还有 N 个任务」，避免用户重复发起。
- 护栏：
  - 上传前按 `Content-Length` 预检磁盘可用空间，不足则拒绝并提示。
  - 单任务大小上限、队列长度上限，超限直接拒绝并提示，不硬扛。

## 8. 后台任务与进度追踪

- 上传合并完成后，后端立即返回 `task_id`，注入转入后台任务，前端不阻塞。
- 任务状态（总数、已完成数、成功/失败明细）**持久化到 SQLite**，不仅存内存。因此：
  - 用户关闭页面后重开，凭 `task_id` 或「我的任务」列表可继续查看进度。
  - 导入服务自身重启后，未完成任务状态不丢失，标记为「中断」，可重试。
- 前端轮询任务状态接口刷新进度条与实时成功/失败计数。
- 任务记录与临时文件在任务结束后保留 7 天供回看，之后自动清理。

任务状态机：`排队中 → 进行中 → 完成 / 失败 / 中断`。

## 9. 错误处理

原则：单点失败不拖垮整批，所有失败汇总报告。

| 场景 | 处理 |
|---|---|
| 上传分片中断 | 后端记录已收分片号，前端续传只补缺失片；临时文件保留至导入完成 |
| tgz 损坏 / 解包失败 | 明确报错「包损坏或格式不支持」，该任务中止，不影响其他任务 |
| 单封 eml 注入失败 | 记录该封（文件名 + 原因），**继续注入后续邮件**，最后汇总失败清单 |
| 完整 tgz 注入失败 | 透传 Zimbra REST 返回的错误信息给前端 |
| 认证失败 / token 过期 | 明确提示重新登录；不向页面直接抛出 Zimbra 内部错误细节 |
| 临时磁盘不足 | 上传前预检；不足则拒绝并提示 |
| 服务重启致任务中断 | 任务标记为「中断」，前端可见，支持重试 |

## 10. 安全

- **全程 HTTPS**：页面收集 Zimbra 账号密码，明文传输不可接受；复用 Zimbra 已有的 Let's Encrypt 证书。
- 导入服务**只监听 `127.0.0.1`**，经 Zimbra nginx 反向代理对外提供，或绑定内网地址；不直接对公网开放。
- 越权防护：普通用户目标账户由后端强制改写为登录账户。
- 上传文件类型限制：仅接受 `.eml` / `.tgz` / `.tar.gz`。
- 临时目录隔离：每个任务独立子目录，任务结束即删；不与 Zimbra 数据目录混放。
- 不落盘用户密码、不落盘任何 authToken、日志不记录凭据；用户登录 token 即用即弃，注入 token 即取即用。
- 服务账号凭据存于服务器本地配置文件，文件权限设为 `600`、属主限定为运行该服务的系统用户；该账号是注入功能的唯一长期凭据，需纳入定期轮换。
- SQLite 任务库不保存任何 token 或密码，仅存目标账户、文件路径、发起人、进度状态。

## 11. 技术栈与部署

- **后端**：Python（兼容 CentOS 7 自带的 python3 版本，预计 3.6；如版本过低则在独立 venv 中部署较新 python）。Web 框架选用轻量、依赖少的方案（如 Flask），避免对 python 新版本的强依赖。
- **解包**：Python 标准库 `tarfile`（原生支持 pax，正确处理 `PaxHeader`）。
- **任务存储**：SQLite（标准库 `sqlite3`，无额外依赖）。
- **前端**：单页，原生 JS（文件切片用 `Blob.slice`，分片并发上传），无重型前端框架。
- **进程模型**：web 进程 + worker 进程，各自由 systemd 管理；通过 SQLite 队列表通信。
- **部署位置**：Zimbra 服务器本机 `192.168.31.175`，独立 venv，独立 systemd 服务。
